# Finishing a Development Branch

**Propósito**: Cerrar limpiamente un branch de desarrollo después de completar todas las tareas.

**Cuándo activar**: Al final de `subagent-driven-development`, cuando todas las tareas están completed y reviewed.

---

## Proceso (5 pasos)

### 1. Verificar tests

```bash
# Correr suite completa
npm test   # o la suite del proyecto
```

❌ Si los tests fallan → NO avanzar. Reportar y pedir acción al usuario.
✅ Si todos pasan → continuar.

### 2. Determinar base branch

```bash
git log --oneline main..HEAD      # commits desde main
git show-branch --sha1-name HEAD main  # base común
```

Identificar de dónde se branché la rama actual.

### 3. Presentar las 4 opciones al usuario

```
✅ Tests: X passing

¿Qué hacemos con feat/<branch-name>?

[1] Merge local    — integrar en <base>, verificar tests, borrar rama
[2] Push + PR      — hacer push y abrir Pull Request
[3] Mantener       — preservar rama y worktree para después
[4] Descartar      — eliminar permanentemente (requiere confirmación)

Elige: [1/2/3/4]
```

### 4. Ejecutar la opción elegida

**Opción 1 — Merge local**:
```bash
git checkout <base-branch>
git merge feat/<branch-name>
npm test   # verificar que sigue pasando
git branch -d feat/<branch-name>
git worktree remove .worktrees/<branch-name>
```

**Opción 2 — Push + PR**:
```bash
git push -u origin feat/<branch-name>
gh pr create --title "<feature name>" --body "<auto-generated summary>"
```
El worktree se preserva.

**Opción 3 — Mantener**:
No hacer nada. Rama y worktree se preservan.

**Opción 4 — Descartar**:
Pedir confirmación explícita: el usuario debe escribir "discard".
```bash
git worktree remove --force .worktrees/<branch-name>
git branch -D feat/<branch-name>
```

### 5. Limpiar worktree

Solo para opciones 1 y 4. Para 2 y 3: preservar worktree.

---

## Post-completion

Actualizar STATE.md:
```
Tarea Activa: ninguna
Tareas Completadas: [feat/<branch-name>] — [opción elegida]
```

---

## Reglas

- ❌ NUNCA hacer merge con tests fallando
- ❌ NUNCA descartar sin la confirmación explícita del usuario
- ✅ Siempre presentar las 4 opciones — no asumir la preferida
- ✅ Limpiar worktree solo en opciones 1 y 4
